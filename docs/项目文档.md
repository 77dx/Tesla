# Tesla 项目文档

## 一、项目概述

Tesla 是一个基于 Django 的**测试平台**，支持 API 接口测试与 UI 测试的用例管理、套件编排和自动化执行。核心能力包括：

- **用例管理**：在数据库中维护接口/UI 用例，按项目组织
- **套件编排**：将多个用例组合成测试套件，支持单次执行、定时任务、WebHook 触发
- **依赖驱动执行**：支持接口间依赖（DAG），自动拓扑排序、分层并发执行
- **报告与隔离**：每次执行独立目录，生成 Allure 报告，支持多套件并行、结果隔离

## 四、核心模块

### 4.1 数据模型关系

```
Project
  ├── Endpoint（接口）
  │     ├── Case（API 用例，一个接口可多个用例）
  │     └── requires / provides（DAG 依赖）
  ├── Suite（测试套件）
  │     ├── case_api_list (M2M → Case)
  │     ├── case_ui_list (M2M → CaseUI)
  │     └── RunResult（执行记录）
  └── Config（项目级 pytest 配置）
```

### 4.2 case_api（API 用例）

- **Endpoint**：接口定义（URL、method、headers、params 等），支持 `requires`/`provides` 声明依赖
- **Case**：接口用例（api_args、extract、validate），通过 `to_yaml()` 生成 apiframetest 格式的 YAML
- **GenerateCase**：从数据库读接口/用例，按 apiframetest 约定生成 YAML 文件

### 4.3 suite（测试套件）

- **Suite**：套件配置，关联 case_api_list / case_ui_list，支持单次、CRON、WebHook
- **RunResult**：一次执行记录，含 path、status、is_pass，报告在 `path/report/`

执行入口：`Suite.run(endpoint_ids=None, initial_context=None)` → 创建 RunResult → 投递 Celery `start_suite_dag`

### 4.4 apiframetest（执行引擎）

- **MainUtil.stand_case_flow**：单用例执行链路  
  `change(request)` → 替换占位符 → 发送请求 → `extract_key` 提取 → 断言
- **extract.yaml / Redis**：用例间变量共享（见后文「extract 变量机制」）
- **tests/conftest.py**：pytest 收集 YAML 文件，生成 `YamlTest` 并执行

## 五、API 接口一览

| 模块 | 路径 | 说明 |
|------|------|------|
| 套件 | `POST /api/suite/{id}/run/` | 执行套件 |
| 套件 | `POST|GET /api/suite/{id}/webhook/?key=xxx` | WebHook 触发 |
| 套件 | `GET /api/suite/result/{id}/` | 查询执行结果 |
| 套件 | `GET /api/suite/static/{path}` | 报告/静态文件（免认证） |
| case_api | `POST /api/case_api/run/` | 单接口执行（endpoint_id） |
| 文档 | `GET /api/schema/swagger/` | Swagger UI |
| 文档 | `GET /api/schema/redoc/` | ReDoc |

### 5.1 执行套件请求体

```json
{
  "endpoint_ids": [1, 2, 3],
  "context": { "token": "预置值" }
}
```

- `endpoint_ids` / `api_ids`：要执行的 Endpoint.id 列表，不传则用套件默认 case_api_list 推导
- `context`：初始上下文，供 requires 无 provider 的变量使用

### 5.2 报告访问

```
http://{host}/api/suite/static/result_{result_id}_{timestamp}/report/index.html
```

## 六、测试执行流程

### 6.1 套件执行（DAG 模式）

```
1. POST /api/suite/{id}/run/ 传入 endpoint_ids
2. Suite.run() 创建 RunResult，目录 upload_yaml/result_{id}_{ts}/
3. start_suite_dag.delay(result_id, endpoint_ids, context)
4. Celery worker:
   - 初始化 Redis 上下文 (suite:{result_id}:context)
   - 构建 DAG（requires/provides）
   - 分层调度：schedule_next_wave → 找到无依赖/依赖已满足的 endpoint
   - execute_endpoint_node：GenerateCase().to_yaml() → pytest(单个 YAML) → 输出 results_raw/{eid}/
   - dag_wave_complete：更新 succeeded/failed，触发下一波
   - finalize_suite_run：合并 results_raw → results，allure generate → report，更新 RunResult
```

### 6.2 单用例执行链路（apiframetest）

```
YAML 用例
  → ExtractUtil.change(request)     # ${yaml_read(xxx)} 替换
  → RequestsUtil.send_all_requests  # 发送请求
  → ExtractUtil.extract_key         # 从响应提取，写入 extract/Redis
  → AssertUtil.assert_all_case      # 断言
```

### 6.3 extract 变量机制

| 步骤 | 说明 |
|------|------|
| 写入 | `extract_key(res, var_name, attr_name, expr, index)` 从响应按 jsonpath/正则提取，写入文件或 Redis |
| 读取 | 请求/断言中 `${yaml_read(userId)}` → `DebugTalk.yaml_read(key)` 从 extract 或 Redis 读取 |
| 后端 | 默认文件 `extract_path`；suite 执行时 `APIFRAME_CONTEXT_BACKEND=redis`，按 `suite:{result_id}:context` 隔离 |

## 七、配置说明

### 7.1 settings.py 主要配置

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| SUITE_EXECUTION_BASE_DIR | 套件执行根目录 | upload_yaml |
| EXTRACT_PATH | 默认 extract 文件路径 | tests/extract.yaml |
| CELERY_BROKER_URL | Celery broker | redis://127.0.0.1:6379/0 |
| CELERY_RESULT_BACKEND | 结果后端 | 同 broker |
| PYTEST_ARGS | pytest 参数 | -v, --tb=short |
| ALLURE_ARGS | allure generate 参数 | --clean |

### 7.2 环境变量

| 变量 | 说明 |
|------|------|
| APIFRAME_EXTRACT_PATH | 覆盖 extract 文件路径 |
| APIFRAME_CONTEXT_BACKEND | file / redis |
| APIFRAME_CONTEXT_PREFIX | Redis 上下文 key 前缀（suite 自动设为 suite:{result_id}） |
| APIFRAME_REDIS_URL | Redis 连接（默认用 CELERY_BROKER_URL） |

## 九、DAG 依赖配置

在 Endpoint 上配置：

- **provides**：该接口会产出的变量名（通常来自 extract）
- **requires**：该接口执行前需要的变量名

示例：登录接口 `provides: ["token", "userId"]`，订单列表 `requires: ["userId"]`，则登录先执行，订单列表依赖登录完成。

## 十、附录：YAML 用例格式

```yaml
feature: 项目名
story: 接口名
title: 用例标题
request:
  url: https://xxx
  method: post
  headers: {}
  data: {}
extract:
  token: [json, $.data.token, 0]
  userId: [json, $.data.userId, 0]
validate:
  equals:
    响应码200: [200, status_code]
```

- extract 格式：`变量名: [attr_name, expr, index]`（attr_name 通常为 `json`，expr 为 jsonpath）
- 引用变量：`${yaml_read(userId)}`
