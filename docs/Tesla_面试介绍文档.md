# Tesla 自动化测试平台 — 面试介绍文档

---

## 一、项目一句话介绍

Tesla 是一个基于 **Django + Vue3** 的企业级接口自动化测试管理平台，支持测试用例的全生命周期管理、多环境配置、测试套件异步执行、以及基于产品线的多租户数据隔离。

---

## 二、项目背景

公司业务系统涉及多个产品线、多套微服务，测试团队需要：
- 统一管理数百个接口和用例，避免散落在本地 Excel/Postman 中
- 支持多环境（测试/预发/生产）切换，不改用例改配置
- 实现 CI/CD 集成，Pipeline 触发自动回归
- 多人协作，按产品线隔离数据，互不干扰

因此自研了 Tesla 平台。

## 三、核心技术栈

| 层级 | 技术选型 | 用途 |
|------|---------|------|
| 后端框架 | Django 4.2 + DRF | REST API 服务 |
| 异步任务 | Celery + Redis | 套件异步执行 |
| 定时任务 | django-q | Cron 定时触发 |

| 状态管理 | Pinia | 用户状态/产品线切换 |
| HTTP | Axios | 前后端通信 |
| 构建工具 | Vite | 前端打包 |

## 四、主要功能模块（面试重点）

### 1. 产品线管理（多租户隔离）
- 整个平台按**产品线**做数据隔离，切换产品线后所有列表自动过滤
- 每个用户可属于多个产品线，每个产品线内有独立的角色权限
- 前端通过 Header 切换器切换当前产品线，后端通过 `product_line` 过滤参数实现隔离

### 2. 接口管理（Endpoint）
- 管理接口基本信息：URL、Method、Headers、Params、Body
- 引入**服务注册表**（Service）概念：接口 URL 由 `{service_key}/path` 组成，service_key 在不同环境中映射不同域名，实现一套用例多环境复用
- 接口声明 `requires`（依赖变量）和 `provides`（产出变量），用于 DAG 调度

### 3. 用例管理（Case）
- 用例与接口解耦：一个接口可以有多个用例（正向/异常/参数化）
- 支持断言配置：equals / contains / startswith 等多种断言类型
- 支持数据提取（extract）：从响应中提取 jsonpath/正则，存入上下文供后续用例使用

### 4. 测试套件执行（核心亮点）
- 套件支持三种触发方式：**单次执行**、**定时任务（Cron）**、**WebHook（CI/CD 集成）**
- 执行时创建**独立目录**，完全隔离，支持多套件并发互不干扰
- 基于 **Celery 异步任务队列**，提交后立即返回，不阻塞用户操作
- 用例支持 **前置(setup) / 正式(main) / 后置(teardown)** 三阶段
- 支持**失败策略**（继续/立即停止）、**重试机制**、**用例超时**配置

### 5. 环境管理
- 支持多服务 URL 配置（微服务场景）
- 支持环境变量、全局请求头、Mock 规则
- 套件执行时选定环境，引擎自动将环境配置注入上下文

### 6. 权限管理
- 基于 RBAC（角色-权限码）模型
- 权限码粒度到操作级别，如 `project:create`、`case:delete`
- 支持产品线级权限覆盖（同一用户在不同产品线可有不同角色）
- 前端路由守卫 + 菜单 v-if 双重鉴权

## 七、可能的面试问题及回答

### Q1: 为什么选择 Celery 做异步执行，而不是 Django 自带的线程？
**A**: Celery 是生产级分布式任务队列，优势有三：
1. **持久化**：任务存入 Redis，Worker 重启后任务不丢失
2. **可扩展**：Worker 可水平扩展，多台机器同时消费任务队列
3. **监控**：可配合 Flower 监控任务状态和执行历史
线程方案在进程重启时任务会丢失，不适合生产环境。

### Q2: 数据隔离是怎么实现的？
**A**: 两个层面：
1. **执行隔离**：每次套件执行创建独立目录 `result_{id}_{timestamp}/`，各套件的用例文件、执行结果、Allure 报告完全分离，互不影响
2. **数据隔离**：基于产品线（ProductLine）模型，Project 关联 ProductLine，所有下游数据（接口/用例/套件/环境）通过 `project__product_line_id` 过滤，不同产品线成员看不到彼此数据

### Q3: 服务注册表解决了什么问题？
**A**: 微服务场景下，一套测试用例需要在多个环境执行。传统做法是把域名硬编码在用例里，换环境需要批量替换。
Tesla 的方案是：接口 URL 只存路径部分，域名由 `service_key` 代替（如 `{user-service}/api/users`），`user-service` 在环境管理中配置为实际域名（如 `https://user.test.example.com`）。切换环境时只改环境配置，用例零修改，完美支持多环境复用。

### Q4: 变量是如何在用例之间传递的？
**A**: 通过「提取-注入」机制：
1. 用例 A 的 `extract` 配置从响应中提取变量（如登录接口提取 token），存入上下文（Context）
2. 用例 B 的参数中用 `${yaml_read(token)}` 引用，执行前引擎自动替换为实际值
3. 上下文存储支持两种后端：本地文件（单机顺序执行）和 Redis（分布式并发执行），套件执行时自动使用 Redis，按 `suite:{result_id}:context` 做 key 隔离

### Q5: 如何防止 RBAC 权限被绕过？
**A**: 三层防护：
1. **后端接口级**：每个 ViewSet action 使用 `RolePermission('code')` 装饰器，管理员自动放行，普通用户检查 `Permission.objects.filter(roles__users=user, code=self.code)` 数据库查询
2. **前端路由守卫**：`router.beforeEach` 检查 localStorage 中的 permissions 数组，没有权限跳转 /403
3. **前端菜单隐藏**：导航菜单用 `v-if="hasPermission('xxx:list')"` 控制显示，避免误操作

### Q6: 套件支持哪些触发方式？
**A**: 三种：
1. **单次执行（Once）**：手动点击运行
2. **定时任务（Cron）**：配置 Cron 表达式，如 `0 2 * * *` 每天凌晨 2 点，由 django-q 调度
3. **WebHook**：创建时自动生成密钥，CI/CD Pipeline 通过 HTTP POST 触发，实现与 Jenkins/GitLab CI 集成

### Q7: 如果执行中的套件失败了，怎么排查？
**A**: 
1. 执行结果页面可以看到 `status`（初始化/准备/运行中/生成报告/完毕/出错）
2. 每次执行有独立目录，里面保留了完整的 Allure 报告
3. 可以点击查看 Allure HTML 报告，精确到每条用例的请求详情、响应内容、断言结果
4. Celery Worker 日志记录了详细的执行过程

### Q8: 项目中遇到的最大挑战是什么？
**A**: 最大挑战是**分布式执行下变量共享**的问题。最初用本地 YAML 文件存储提取变量，单机顺序执行没问题，但多套件并发时多个 Worker 进程读写同一文件会出现竞争条件，导致变量值错乱。
解决方案是引入 Redis 作为上下文存储后端，每个套件执行用独立的 Redis key（`suite:{result_id}:context`），Worker 通过原子操作读写，彻底解决了并发冲突问题。

### Q9: 项目规模和数据量大概是多少？
**A**: 当前平台管理约 XXX 个接口、XXX 条用例，支持 XX 个产品线同时使用，每日执行约 XX 次套件，平均每次执行 XX 条用例，执行时间约 XX 分钟（根据实际数据填写）。

### Q10: Vue3 和 Vue2 的区别，为什么选 Vue3？
**A**: 选 Vue3 主要因为：
1. **Composition API**：逻辑复用更好，可以把相关逻辑组合在一起而不是分散在 data/methods/computed 中
2. **Pinia 状态管理**：比 Vuex 更简洁，天然支持 TypeScript
3. **性能**：响应式系统基于 Proxy，比 Vue2 的 Object.defineProperty 性能更好，且能监听数组/动态属性变化
4. **生态**：Vite 构建速度极快，开发体验好

> "Tesla 是我参与开发的企业级接口自动化测试管理平台。技术上，后端基于 Django + DRF，前端 Vue3，异步执行用 Celery + Redis，测试报告集成 Allure。

> 我在这个项目中负责了从数据库模型设计到前端页面的全栈开发工作。"
